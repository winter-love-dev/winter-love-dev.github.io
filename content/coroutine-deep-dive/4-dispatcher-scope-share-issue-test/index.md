# 채널

코루틴끼리 통신을 위한 기본적인 방법.

코루틴 1 `(송신)` -> 채널 -> `(수신)` 코루틴 2 

이런 구조로 서로 다른 코루틴 1, 2 가 채널을 통해 송신, 수신을 하는 구조이다.

`SendChannel`, `ReceiveChannel` 
두 개의 인터페이스를 구현한게 우리 프로젝트에서도 사용하는 `Channel` 이다.

아래와 같은 동작 특징이 있다.
- `receive` 를 호출했을 때 채널에 원소가 없다면 코루틴은 원소가 들어올 때까지 중단된다.
- `send` 는 채널의 용량이 다 찼을때 중단된다.

채널은 송신자(provider?)와 수신자(receiver) 수에 제한이 없다.
하지만 송신, 수신자 양 끝에 하나의 코루틴만 있는 경우가 일반적이다.

* 채널을 통해 단 한 번만 값을 수신할 수 있다 (중요)
* `send`, `receive`
 
## 채널 타입

**무제한**
제한이 없는 용량 버퍼를 가진 `Channel.UNLIMITED` 타입.
send 가 중단되지 않는다.

**버퍼**
특정 용량 크기로 설정된 채널 `Channel.BUFFERED` 타입.
기본 버퍼 값은 64로 설정되어 있다.

**랑데뷰**
용량이 0이거나, `Channel.REENDEZYOUS` 타입인 채널로 송신자와 수신자가 만날때만 원소를 교환한다.
데이터가 채널을 통해 거쳐가는것은 맞지만
서로 다른 코루틴이 직접 데이터를 교환하는 느낌에 가깝다.

**융합**
`Channel.CONFLATED` 타입을 가진 채널로, 새로운 원소가 이전 원소를 대체한다.

## 버퍼 오버플로일 때
채널을 커스텀화 하려면
버퍼가 가득 찼을때의 행동을 정의할 수 있다.

`BufferOverflow.SUSPENDED`
버퍼가 가득 찼을때 send 메소드가 중단된다.

`BufferOverflow.DROP_OLDEST`
버퍼가 가득 찼을 때, 가장 오래된 원소가 제거된다.

`BufferOverflow.DROP_LATEST`
버퍼가 가득 찼을 때, 가장 최근의 원소가 제거된다.

## 전달되지 않은 원소 핸들러

`Channel` 함수에서 `onUndeliveredElement` 라는 파라미터는 반드시 알아야 한다.

원소가 어떠한 이유로 처리되지 않을때 호출된다.

대부분 채널이 닫히거나 취소됨을 의미한다.

에러를 던질때 발생할 수도 있다.

## 팬아웃

여러개의 코루틴이 하나의 채널로부터 원소를 받을 수도 있다.

이런 경우 원소를 적절하게 처리하려면 반복문 루프를 사용해야 한다.

3 개의 코루틴이 하나의 채널로부터 원소를 받는다고 가정해보자.

반복문을 통해 receive 를 할 경우, 

모든 코루틴은 공평하게 1, 2, 3 이라는 값을 받을수 있게 된다.

* 채널을 통해 단 한 번만 값을 수신할 수 있다 (중요)
* 내용 틀림 다시 정리

## 팬인

여러 개의 코루틴이 하나의 채널로 원소를 전송할 수 있다.

다수의 채널을 하나의 채널로 합쳐야 할 경우도 있다.

이런 경우 produce 함수로 여러 개의 채널을 합치는 fanIn 함수를 사용할 수 있다.

## 파이프라인

한 채널로부터 받은 원소를 다른 채널로 전송하는 경우에 사용한다.
이것을 파이프라인이라고 부른다.

## 통신의 기본 형태로서의 채널

지금까지 알아본 채널의 활용 예시를 보여주는 장이다.

produce 를 활용해서 파이프라인을 사용하는 예시와
fanIn 을 활용해서 코루틴들이 생성한 결과를 하나로 합치는 예시를 보여주고 있다.

# 셀렉트

이번 장에서 "경합" 이라는 키워드를 자주 사용하는데
국어사전상 "서로 맞서 겨룸" 이라고 한다.

`select` 라는 코루틴 함수가 있다.
가장 먼저 완료되는 코루틴의 결과를 기다리는 역할을 한다.

여러 개의 채널 중, 
이용 가능한 원소가 있는 채널로부터 
데이터를 받을 수 있는지 여부도 확인할 수 있다.

## 지연되는 값 선택하기

여러 개의 소스에 데이터를 요청한 뒤, 가장 빠른 응답만 얻는 경우가 있다고 생각해보자.

여러 개의 비동기 프로세스로 시작한 뒤, `select` 함수를 표현식으로 사용하고 내부에서 값을 기다리는 방법이 있다.

`select` 내부적으로는 셀렉트 표햔식에서 나올수 있는 결과 값을 명시하는 `Deferred` 값의 `onWait` 함수를 호출한다.

람다식 내부에서 값을 변환할 수도 있다.

책에서 두 개의 비동기 작업을 `select` 블록에서 실행한 뒤 
`onAwait {}` 블록을 감싸서 호출 결과를 기다리는 예제를 보여주고 있다.
data1 과 data2 요청이 있는데, 응답이 더 빠른 data2 의 결과값을 출력하고 코드는 종료된다.

## 채널에서 값 선택하기

select 함수는 채널에서도 사용할 수 있다.

셀렉트 표현식에서 사용하는 주요 함수:

`onReceive`
채널이 값을 가지고 있을때 선택된다.
람다식의 인자로 먼저 넘어오는 값이 select 의 반환 값이 된다.

`onReceiveCatching`
채널이 값을 가지고 있거나 닫혔을 때 선택된다.
ChannelResult 를 통해 값을 밥거나 채널이 닫혔다는 이벤트를 받을수 있다.
이 값을 람다식의 인자로 사용한다.
`onReceiveCatching` 이 선택 되었을 때 select 는 람다식의 결과 값을 반환한다.

`onSend`
채널의 버퍼에 공간이 있을때 선택된다.
채널에 값을 보낸 뒤, 채널의 참조 값으로 람다식을 수행한다.

# 핫 데이터 소스와 콜드 데이터 소스

채널은 핫 스트림이다.
하지만 콜드 스트림이 필요할때가 있다.

List, Set 과 같은 컬렉션은 핫이고
Sequence 나 자바의 Stream 은 콜드이다.

채널은 핫이지만
Flow 와 RxJava 스트림은 콜드이다.

## 핫 VS 콜드

핫 데이터 스트림은 데이터를 소비하는것과 무관하게 항상, 즉각 실행된다.

콜드 스트림은 요청이 있을때만 실행되며, 아무것도 저장하지 않는다.

핫의 예시로 컬렉션으로 반복문 연산을 3회 돌리면 즉각 모든 연산을 처리한다.

콜드의 예시로 반복문 연산을 3회 돌린다고 가정했을 때, 
`println` 같이 값을 조회하려 하는 등 일종의 소비를 하려 하는 순간에만 연산 처리가 된다.

이러한 특성으로 콜드 스트림은 
- 무한할 수 있다.
- 최소한의 연산만 수행한다.
- (왜냐하면) 메모리를 적게 사용한다.

핫 데이터 스트림은 
- 항상 사용 가능한 상태이다 (각 연산이 최종 연산이 될 수 있다)
- 여러 번 사용되었을 때 매번 결과를 다시 계산할 필요가 없다.

## 핫 채널, 콜드 플로우

채널(핫)과 플로우(콜드)의 동작방식 차이를 설명하는 장이다.

플로우를 생성하는 일반적인 방법은 `produce` 함수와 비슷한 형태의 빌더를 사용하는 것이다.
이 빌더가 바로 `flow` 이다.

```
val channel = produce {
  while (true) {
    val x = computeNextValue()
    send(x)
  }
}

val flow = flow {
  while (true) {
    val x = computeValue()
    emit(x)
  }
}
```

두 빌더는 개념적으로 동일하다.
하지만 채널과 플로우는 방식이 아주 다르다.

produce 는 일종의 코루틴 빌더이기 때문에 코루틴 스코프 안에서 실행될 수 있다.
그리고 이것은 핫 스트림이기 때문에 소비 여부와 상관없이 값을 생성한다.

flow 는 빌더가 아니며, 어떤 처리도 하지 않는다.
`collect` 와 같은 최종 연산이 호출될 때(소비되는 타이밍) 원소가 어떻게 생성되어야 하는지 정의한 것에 불과하다.
이러한 이유로 flow 는 CoroutineScope 가 필요하지 않다.

`collect` 와 같은 최종 연산 지점에선 `coroutineScope` 가 필요함.

## 정리

대부분의 데이터 소스는 핫이거나 콜드이다.

핫은 가능한 빨리 원소를 만들고 저장한다.
원소가 소비되는것과 무관하게.
List, Set 과 같은 컬렉션이나 Channel 이 핫 데이터 소스에 해당한다.

콜드는 최종 연산에서 값이 필요할 때가 되어서야 처리한다.
중간 과정의 모든 함수는 무엇을 해야 할지만 정의 해둔것에 불과하다.
일반적으로 원소를 저장해두지 않으며, 필요할 때 원소를 생성한다.
연산은 최소한으로 수행하며, 무한정일 수 있다.
